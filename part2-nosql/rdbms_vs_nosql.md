## Database Recommendation
For a patient management system, I would recommend MySQL over MongoDB.
The main reason is that in healthcare, data has to be accurate at all times. If a doctor is updating a patient's medication and the system crashes halfway through, we cannot have half the update saved and half missing. MySQL handles this through ACID properties, which makes sure every transaction either completes fully or gets rolled back. This kind of reliability is really important when lives are involved.
MongoDB works on BASE principles, which means it is okay with data being "eventually consistent." That is fine for something like an online store where showing a slightly outdated product price is not a big deal. But in a hospital, wrong data at the wrong time can cause serious harm.
The CAP theorem also supports this choice. MongoDB prioritizes availability, meaning it will keep responding even during network problems, but the data it returns might not be fully updated. MySQL prioritizes consistency over availability, which is the right tradeoff for patient records.
Also, patient data is very structured. Every patient has a name, age, blood group, diagnosis, and assigned doctor. A fixed relational schema with proper foreign keys makes sure this data stays clean and connected. MongoDB's flexible schema is useful when you don't know what your data will look like, but here we do know.
If the system also needs a fraud detection module, then I would bring MongoDB in for that part. Fraud detection deals with large amounts of activity logs and real-time event data, which is exactly where MongoDB performs well. So the best approach would be to use MySQL for core patient records and MongoDB for the fraud detection pipeline. Both databases doing what they are actually good at.
